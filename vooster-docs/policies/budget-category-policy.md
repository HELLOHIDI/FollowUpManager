# 3. 비목별 기본 정책 공통 구조 v1.1

## 3-1. 정책 목적

비목별 기본 정책은 GrantFollow에서 정부지원사업비 지출을 목적별 예산 항목으로 분류하고, 사용자가 지출을 등록하거나 수정할 때 선택한 비목에 따라 필요한 기본 증빙서류, 유의사항, 사용 가능 범위를 일관된 형식으로 안내하기 위해 정의한다.

비목은 지출의 분류 기준이다.
비목 자체는 단계, 승인 상태, 수행 상태, 집행 신청 상태, 증빙 파일을 직접 가지지 않는다.

GrantFollow에서 단계, 금액, 거래처, 증빙, 승인, 수행, 집행 신청 정보는 모두 개별 지출에 귀속된다.
비목별 기본 정책은 사용자가 특정 지출에 비목을 선택했을 때, 해당 비목 기준으로 어떤 서류와 유의사항을 확인해야 하는지 안내하기 위한 정책 템플릿이다.

이 정책은 지출을 자동 판정하거나 정산 적합성을 최종 판단하기 위한 규칙 엔진이 아니다.
MVP 단계에서는 사용자가 지출 등록 또는 수정 화면에서 비목을 선택했을 때 해당 비목의 기본 서류와 주의사항을 빠르게 확인할 수 있도록 돕는 “정책 안내 템플릿”으로 사용한다.

따라서 이 섹션에서는 아래 내용만 다룬다.

```text
- 비목명
- 비목 정의
- 사용 가능 범위
- 기본 증빙 요건
- 조건부 증빙 요건
- 유의사항
- 집행 제한 또는 사전 검토가 필요할 수 있는 항목
- 화면 안내 문구
- Codex 개발 기준
```

아래 내용은 이 섹션에서 다루지 않는다.

```text
- 사업별 비목 예산
- 특정 사업의 지출 내역
- 지출 단계
- 지출별 검증 결과
- 준비도 계산
- risk flag 계산
- evaluation snapshot
- 대시보드 KPI 계산
```

## 3-2. 정책 적용 범위

비목별 기본 정책은 다음 상황에서 사용된다.

```text
1. 사용자가 지출을 등록하거나 수정할 때 비목을 선택하는 경우
2. 선택된 비목에 따라 지출 상세 풀페이지에서 기본 증빙서류를 안내하는 경우
3. 선택된 비목에 따라 지출 상세 풀페이지에서 조건부 증빙서류를 안내하는 경우
4. 선택된 비목에 따라 지출 상세 풀페이지에서 유의사항을 안내하는 경우
5. 지출에 증빙 파일을 업로드할 때, 선택된 비목 기준으로 필요한 서류를 안내하는 경우
6. 향후 비목별 정책 seed 데이터를 생성하는 경우
```

MVP에서는 비목별 기본 정책이 사용자의 입력을 강제로 차단하지 않는다.
비목별 기본 정책은 “이 비목으로 분류된 지출에는 보통 이런 서류와 주의사항이 필요하다”는 기준 정보를 제공한다.

---

## 3-3. 비목 정책 해석 기준

비목별 기본 정책은 아래 자료를 참고해 작성한다.

```text
1. 창업사업화 지원사업 통합관리지침
2. 초기창업패키지 창업기업등 사업비 비목 해설표
3. GrantFollow MVP 운영 목적
```

단, 실제 사업비 집행 시에는 아래 기준이 우선될 수 있다.

```text
1. 개별 사업 협약서
2. 사업별 공고문 및 세부관리기준
3. 주관기관 또는 전문기관 안내
4. 창업사업화 지원사업 통합관리지침
5. 초기창업패키지 등 사업별 비목 해설표
6. GrantFollow 기본 정책 템플릿
```

따라서 GrantFollow의 기본 정책은 공식 지침을 대체하지 않는다.
사용자가 사업별 유의사항을 별도로 입력할 수 있도록 열어두어야 한다.


## 3-5. 비목 기본 정책 템플릿 구조

각 비목 정책은 아래 공통 구조를 가진다.

```ts
type BasicPolicyTemplate = {
  category_key: BudgetCategoryKey;
  category_name: string;
  category_description: string;
  usage_scope: string;

  evidence_requirements: EvidenceRequirement[];
  caution_notes: PolicyCautionNote[];
  restricted_notes: PolicyRestrictedNote[];

  ui_guide_message: string;
  evidence_guide_message: string;
  caution_guide_message: string;

  source_type: PolicySourceType;
  source_title: string;
  source_version?: string | null;

  is_active: boolean;
  version: number;
};
```

---

## 3-6. 필드별 설명

### category_key

개발에서 비목을 식별하기 위한 고유 key다.

예시:

```text
outsourcing_cost
```

### category_name

사용자 화면에 표시되는 비목명이다.

예시:

```text
외주용역비
```

### category_description

비목의 정의를 설명한다.

예시:

```text
창업아이템을 고도화하거나 사업계획을 수행하기 위해 일부 공정을 외부 업체에 의뢰하고 대가를 지급하는 비용
```

### usage_scope

해당 비목으로 분류할 수 있는 지출의 범위를 설명한다.

예시:

```text
사업계획서상 창업아이템 또는 과제 수행과 직접 관련된 제작, 개발, 디자인, 시험, 검수 등 외부 용역 지출을 이 비목으로 분류할 수 있다.
```

## 3-7. 증빙 요건 구조

증빙서류는 단순 문서 목록이 아니라 `EvidenceRequirement` 단위로 관리한다.

이유는 “세금계산서 또는 신용카드 영수증”처럼 둘 중 하나만 있어도 충족되는 증빙이 있고, “검수조서와 증빙사진”처럼 둘 다 필요한 증빙이 있기 때문이다.

단, MVP에서는 이 구조를 자동 검증 엔진으로 사용하지 않는다.
이 구조는 지출 상세 풀페이지와 증빙 업로드 영역에서 사용자에게 필요한 서류를 안내하기 위한 기준이다.

```ts
type EvidenceRequirement = {
  requirement_key: string;
  requirement_name: string;

  requirement_type: "required" | "conditional" | "optional";

  fulfillment_type: "any_of" | "all_of";

  accepted_documents: EvidenceDocumentOption[];

  condition_label?: string | null;
  description?: string | null;
  upload_guide_message?: string | null;

  sort_order: number;
};
```

### requirement_type

```text
required
- 해당 비목으로 분류된 지출에서 기본적으로 필요한 증빙

conditional
- 특정 조건이 있을 때 필요한 증빙

optional
- 있으면 참고가 되는 증빙
```

### fulfillment_type

```text
any_of
- accepted_documents 중 하나만 첨부되어도 충족되는 형태

all_of
- accepted_documents 전체가 첨부되어야 충족되는 형태
```

---

## 3-8. 증빙 문서 옵션 구조

```ts
type EvidenceDocumentOption = {
  document_key: string;
  document_name: string;
  description?: string | null;
};
```

### 예시 1. 거래 증빙

세금계산서 또는 신용카드 영수증 중 하나로 충족되는 구조다.

```ts
const transactionProofRequirement: EvidenceRequirement = {
  requirement_key: "transaction_proof",
  requirement_name: "거래 증빙",
  requirement_type: "required",
  fulfillment_type: "any_of",
  accepted_documents: [
    {
      document_key: "tax_invoice",
      document_name: "세금계산서",
    },
    {
      document_key: "credit_card_receipt",
      document_name: "신용카드 영수증",
    },
  ],
  description: "세금계산서 또는 신용카드 영수증 중 하나를 첨부합니다.",
  upload_guide_message: "거래 사실과 금액을 확인할 수 있는 증빙을 첨부해주세요.",
  sort_order: 1,
};
```

### 예시 2. 검수 증빙

검수조서와 증빙사진이 함께 필요한 구조다.

```ts
const inspectionProofRequirement: EvidenceRequirement = {
  requirement_key: "inspection_proof",
  requirement_name: "검수 증빙",
  requirement_type: "required",
  fulfillment_type: "all_of",
  accepted_documents: [
    {
      document_key: "inspection_report",
      document_name: "검수조서",
    },
    {
      document_key: "evidence_photo",
      document_name: "증빙사진",
    },
  ],
  description: "검수조서와 증빙사진을 함께 첨부합니다.",
  upload_guide_message: "납품 또는 수행 완료를 확인할 수 있는 검수자료를 첨부해주세요.",
  sort_order: 3,
};
```

## 3-9. 조건부 증빙 표현 방식

MVP에서는 복잡한 조건 엔진을 만들지 않는다.
조건부 증빙은 우선 `condition_label`이라는 텍스트 필드로 관리한다.

예시:

```ts
const comparativeQuoteRequirement: EvidenceRequirement = {
  requirement_key: "comparative_quote",
  requirement_name: "비교견적서",
  requirement_type: "conditional",
  fulfillment_type: "all_of",
  accepted_documents: [
    {
      document_key: "comparative_quote",
      document_name: "비교견적서",
    },
  ],
  condition_label: "2천만원 기준 거래 시 필요할 수 있음",
  description: "사업별 세부관리기준에 따라 2천만원 기준 거래 시 비교견적서가 필요할 수 있습니다.",
  upload_guide_message: "거래금액이 2천만원 기준에 해당하는 경우 비교견적서 필요 여부를 확인해주세요.",
  sort_order: 99,
};
```

MVP에서는 조건부 증빙을 자동 활성화하지 않아도 된다.
우선 사용자가 확인할 수 있도록 정책 카드에 표시한다.

향후 고도화 단계에서 `condition_label`을 `trigger_condition` 구조로 확장할 수 있다.

## 3-10. 유의사항 구조

비목별 유의사항은 사용자가 지출 전 확인해야 하는 안내 문구다.

```ts
type PolicyCautionNote = {
  note_key: string;
  note: string;
  severity: "info" | "warning" | "danger";
  sort_order: number;
};
```

### severity 기준

```text
info
- 일반 안내

warning
- 정산 리스크가 있을 수 있어 확인이 필요한 내용

danger
- 집행 제한 가능성이 있어 특히 주의해야 하는 내용
```

예시:

```ts
const outsourcingAdvancePaymentNote: PolicyCautionNote = {
  note_key: "outsourcing_advance_payment_limit",
  note: "선급금은 외주용역 계약 총액의 50% 이하로 집행해야 합니다.",
  severity: "warning",
  sort_order: 1,
};
```

## 3-11. 집행 제한 또는 사전 검토 안내 구조

비목별로 집행 제한 또는 사전 검토가 필요할 수 있는 항목은 별도 구조로 관리한다.

```ts
type PolicyRestrictedNote = {
  note_key: string;
  note: string;
  severity: "warning" | "danger";
  resolution_guide?: string | null;
  sort_order: number;
};
```

이 항목은 “무조건 집행 불가”를 의미하지 않는다.
사업별 기준, 협약서, 주관기관 안내에 따라 달라질 수 있는 검토 필요 항목을 사용자에게 알려주는 목적이다.

예시:

```ts
const advertisingGiftRestrictedNote: PolicyRestrictedNote = {
  note_key: "advertising_gift_restricted",
  note: "일회성 무작위 홍보를 위한 기념품, 유니폼, 기프티콘 등은 집행 제한 가능성이 높습니다.",
  severity: "danger",
  resolution_guide: "홍보 목적, 집행 결과물, 광고 집행 내역을 확인할 수 있는 자료를 함께 검토하세요.",
  sort_order: 1,
};
```

## 3-12. 정책 출처 구조

```ts
type PolicySourceType =
  | "integrated_guideline"
  | "program_specific_guideline"
  | "grantfollow_default";
```

### source_type 정의

```text
integrated_guideline
- 창업사업화 지원사업 통합관리지침 등 공통 지침 기반

program_specific_guideline
- 초기창업패키지 등 사업별 비목 해설표 기반

grantfollow_default
- GrantFollow가 사용자 편의를 위해 구조화한 기본 정책
```

MVP에서는 사용자 커스텀 정책을 이 구조에 포함하지 않는다.
사용자 입력 메모는 사업별 유의사항 또는 지출 메모로 별도 관리한다.

## 3-13. 정책 화면 표시 기준

비목별 기본 정책은 아래 화면에 표시된다.

```text
1. 지출 빠른 등록 페이지의 비목 선택 도움말
2. 지출 상세 풀페이지의 비목 정책 패널
3. 지출 상세 풀페이지의 증빙 업로드 영역
4. 비목별 정책 안내 카드
```

### 지출 빠른 등록 페이지

```text
- 비목 선택 드롭다운
- 선택된 비목의 한 줄 정의
- 주요 유의사항 1~2개 요약
```

빠른 등록 페이지는 영업 완료 카드 생성을 위한 최소 입력 화면이므로, 긴 정책 설명과 전체 증빙 목록을 모두 펼치지 않는다.
상세한 정책과 증빙 요건은 지출 상세 풀페이지에서 확인한다.

### 지출 상세 풀페이지의 비목 정책 패널

```text
- 선택한 비목의 정의
- 선택한 비목의 사용 가능 범위
- 해당 비목으로 분류된 지출의 기본 증빙
- 해당 비목으로 분류된 지출의 조건부 증빙
- 해당 비목으로 분류된 지출의 유의사항
- 집행 제한 또는 사전 검토가 필요할 수 있는 항목
```

비목 정책 패널은 지출 상세 풀페이지의 우측 고정 패널 또는 정책 안내 섹션에 표시할 수 있다.
비목을 변경하면 정책 안내와 증빙 요건은 변경된 비목 기준으로 다시 표시한다.
단, 이미 업로드된 증빙 파일은 개별 지출에 연결된 상태를 유지하고, 새 비목 기준 충족 여부만 다시 계산한다.

### 지출 증빙 업로드 영역

```text
- 선택된 비목 기준 증빙 요건명
```

## 3-15. Codex 개발 기준

Codex는 비목별 기본 정책을 구현할 때 아래 기준을 따른다.

```text
- 이 문서는 비목 정책 원본 데이터만 정의한다.
- 비목은 지출의 분류 기준이며 실행 상태를 가지지 않는다.
- 비목은 단계, 승인 상태, 수행 상태, 집행 신청 상태, 증빙 파일을 직접 가지지 않는다.

- budget-category-policy.md는 지출 등록 시 선택된 비목에 따라 표시할 안내 정보를 정의한다.

- 사업별 예산, 지출 검증, 준비도 계산, risk flag 계산은 이 섹션에서 구현하지 않는다.

- 각 비목 정책은 category_key로 식별한다.
- 각 지출은 선택된 비목을 참조하기 위해 budget_category_id 또는 category_key를 가진다.
- 증빙은 EvidenceRequirement 구조로 관리한다.
- 세금계산서 또는 신용카드 영수증은 any_of 구조로 표현한다.
```

실제 집행 기준은 개별 사업의 협약서, 공고문, 세부관리기준, 주관기관 또는 전문기관 안내를 우선한다.

```text
- 2천만원 기준 거래 시 비교견적서가 필요할 수 있음
- 광고 선전을 위해 외주용역을 진행하는 경우 외주용역비 기준을 함께 확인해야 할 수 있음
- 온라인 광고, 문자 발송 등 충전·차감 방식 지출은 협약기간 내 홍보 완료 증빙이 필요할 수 있음
```

### 유의사항

```text
- 광고 또는 홍보 활동은 제품·기업 홍보 목적과 직접 관련되어야 한다.
- 홍보 제작물, 광고 집행 결과, 캠페인 결과물을 확인할 수 있어야 한다.
- 온라인 광고나 문자 등 충전·차감 방식은 협약기간 내 실제 홍보가 완료된 비용인지 확인해야 한다.
- 광고 제작을 외주로 진행하는 경우 계약서와 결과보고서 관리가 중요하다.
```

### 집행 제한 또는 사전 검토가 필요할 수 있는 항목

```text
- 일회성 무작위 홍보를 위한 배포용 기념품 제작
- 유니폼 제작
- 기프티콘 등 단순 배포성 비용
- 협약기간 이후 사용할 수 있는 잔여 충전금 또는 이월성 전자화폐
- 결과물 또는 광고 집행 내역을 확인하기 어려운 마케팅 비용
```

### 화면 안내 문구

```text
광고선전비는 제품과 기업을 홍보하기 위한 마케팅 지출을 분류하는 비목입니다.
해당 비목으로 지출을 등록할 경우 계약서, 결과물, 홍보제작물, 광고 집행 증빙을 함께 준비하고 단순 배포성 비용은 주의해주세요.
```
## Expense-derived category contract (2026-06-23)

`project_budget_categories` classifies expenses; it does not hold an independently allocated category budget. Active category planned total is derived from active `expenses.amount` rows, and completed category spend is the `execution_completed` subset. Groups exist only when active expenses use an active, non-deleted category whose active policy template supplies the Korean display name. Missing or inactive metadata is an integrity error. Legacy `budget_amount` values are preserved exactly in the locked, non-authoritative `project_budget_category_budget_archive` during migration 0015 and are not used by active reads.
