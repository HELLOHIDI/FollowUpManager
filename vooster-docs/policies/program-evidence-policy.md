# Program Evidence Policy

상태: 2026-06-29 기능 기획 기준
적용 범위: 사업별 집행 증빙서류 PDF를 기준으로 비목, 하위 유형, 증빙 요구사항을 생성하고 지출 카드에 적용하는 정책

## 1. 목적

사업별 집행 증빙서류 문서는 사업마다 형식과 비목 구성이 다르다.
따라서 이 기능은 기존 공통 비목 정책을 강제로 적용하지 않고, 사업 등록 시 첨부한 정책 문서를 기준으로 해당 사업의 비목과 증빙 요구사항을 만든다.

MVP 목표는 PDF 내용을 자동으로 최종 판정하는 것이 아니다.
시스템은 PDF에서 정책 초안을 만들고, 관리자가 검수하고 확정한 정책만 지출 등록과 지출 상세 화면에 적용한다.

## 2. 기본 원칙

- 정책 파일이 없는 사업은 현재 기존 비목 선택 방식과 지출 카드 레이아웃을 유지한다.
- 정책이 아직 확정되지 않은 사업도 지출 생성은 허용한다.
- 정책 미확정 상태에서는 사업별 비목과 증빙 자동 세팅을 적용하지 않는다.
- 정책 확정 후 기존 지출에는 사용자가 선택해서 정책을 적용하거나 갱신한다.
- 확정된 정책은 직접 수정하지 않고, 변경이 필요하면 새 버전을 만든다.
- 기존 지출은 생성 또는 비목 변경 시점의 정책 버전 스냅샷을 참조한다.

## 3. 정책 생성 흐름

```text
사업 등록 PDF 첨부
-> PDF 자동 추출 시도
-> 비목, 하위 유형, 증빙 요구사항 초안 생성
-> 관리자 검수 및 수정
-> 정책 확정
-> 지출 카드와 지출 상세에 자동 적용
```

추출 실패 또는 불완전한 추출은 오류로 종료하지 않는다.
해당 정책 초안은 검수 필요 상태로 두고 관리자가 직접 보정할 수 있게 한다.

## 4. 정책 상태

```ts
type ProgramEvidencePolicyStatus =
  | "draft_extracted"
  | "needs_review"
  | "ready_to_confirm"
  | "confirmed"
  | "archived";
```

- `draft_extracted`: PDF에서 자동 추출된 초안이다.
- `needs_review`: 추출 실패 또는 애매한 항목이 있어 검수가 필요하다.
- `ready_to_confirm`: 관리자가 필수 항목을 정리했고 확정할 수 있다.
- `confirmed`: 지출 카드와 지출 상세에 적용할 수 있는 확정 정책이다.
- `archived`: 더 이상 새 지출에 사용하지 않는 이전 버전이다.

## 5. 비목과 하위 유형

사업별 정책은 문서에서 추출한 비목을 기준으로 만든다.
기존 기본 9개 비목은 정책 파일이 없는 사업의 fallback UI에서만 사용한다.

지출 분류는 아래 구조를 따른다.

```ts
type ProgramBudgetCategory = {
  category_key: string;
  category_name: string;
  sort_order: number;
  subcategories: ProgramBudgetSubcategory[];
};

type ProgramBudgetSubcategory = {
  subcategory_key: string;
  subcategory_name: string;
  sort_order: number;
};
```

- 상위 비목은 지출 등록 시 필수 선택이다.
- 하위 유형은 정책 문서에 있을 때만 선택한다.
- 예: `지급수수료 > 멘토링비`, `교육훈련비 > 문헌구입비`.
- 하위 유형이 없는 비목은 상위 비목만 선택한다.

stable key는 시스템이 표시명을 기준으로 자동 생성한다.
관리자는 정책 확정 전까지만 key를 수정할 수 있으며, 확정 후에는 key를 바꿀 수 없다.

key 생성과 수정 규칙:

- 한글 원문명과 화면 표시명은 그대로 보존하고, key는 별도 값으로 관리한다.
- key는 ASCII slug 형식을 사용한다. 예: `education_training`, `education_training_2`.
- 같은 정책 버전 안에서 `category_key`는 중복될 수 없다.
- 같은 상위 비목 안에서 `subcategory_key`는 중복될 수 없다.
- 증빙 요구사항 key도 같은 정책 버전 안에서 중복될 수 없다.
- 자동 생성한 key가 충돌하면 suffix를 붙인다.
- 관리자가 key를 직접 수정할 때도 slug 형식 검증과 중복 검사를 통과해야 한다.
- 지출 스냅샷과 대시보드 필터는 표시명이 아니라 확정된 key를 기준으로 연결한다.

## 6. 증빙 요구사항 적용

증빙 목록은 상위 비목 공통 증빙과 하위 유형 증빙을 합쳐서 표시한다.

```text
지출 증빙 요구사항 =
  선택한 상위 비목의 공통 증빙
  + 선택한 하위 유형의 추가 증빙
```

동일한 증빙이 공통과 하위 유형에 모두 있으면 하나로 병합한다.
병합 기준은 표준 문서 키 또는 사업별 문서 키이며, 원문 문구는 모두 보존한다.

지출은 정책을 단순 동적 참조하지 않는다.
지출 생성 또는 비목/하위 유형 변경 시점에 확정 정책 버전과 증빙 요구사항을 스냅샷으로 저장한다.

```ts
type ExpensePolicySnapshot = {
  policy_version_id: string;
  category_key: string;
  category_name: string;
  subcategory_key?: string | null;
  subcategory_name?: string | null;
  evidence_requirements_snapshot: EvidenceRequirement[];
};
```

- `policy_version_id`는 당시 적용한 확정 정책 버전을 가리킨다.
- `category_key`와 `subcategory_key`는 당시 선택한 정책 항목의 stable key다.
- 화면의 증빙 목록은 `evidence_requirements_snapshot`을 기준으로 표시한다.
- 정책 원문과 근거는 `policy_version_id`를 통해 추적한다.
- 새 정책 버전이 확정되어도 기존 지출의 스냅샷은 자동 변경하지 않는다.
- 지출의 비목 또는 하위 유형을 바꾸면 해당 시점의 확정 정책 기준으로 새 스냅샷을 만든다.

1차에서는 증빙 요구사항을 자동 검증하거나 지출 저장/단계 이동을 차단하지 않는다.
업로드된 증빙 파일과 정책 요구사항은 document key 기준으로 느슨하게 매칭하여 상태만 표시한다.

```ts
type EvidenceRequirementFulfillmentStatus =
  | "not_uploaded"
  | "uploaded"
  | "waived";
```

- `not_uploaded`: 요구사항에 연결된 증빙 파일이 아직 없다.
- `uploaded`: 요구사항에 연결된 증빙 파일이 1개 이상 있거나 충족 방식 기준을 만족한다.
- `waived`: 관리자가 해당 없음 또는 제외로 수동 처리했다.

충족 상태 표시 규칙:

- `required`, `conditional`, `optional`은 차단 조건이 아니라 배지와 안내 문구로 표시한다.
- `conditional`은 자동 판단하지 않고 조건 문구를 그대로 보여준다.
- `any_of`는 후보 증빙 중 하나라도 업로드되면 `uploaded`로 표시한다.
- `all_of`는 후보 증빙이 모두 업로드되어야 `uploaded`로 표시한다.
- `waived`는 관리자 수동 상태로만 설정할 수 있다.
- 부족한 증빙이 있어도 경고와 남은 항목 안내만 표시하고 지출 저장이나 단계 이동을 막지 않는다.

## 7. 증빙 문서 키 기준

사업별 정책은 자체 증빙 문서 목록을 갖는다.
다만 여러 사업에서 반복되고 제출 의미가 안정적인 문서는 표준 문서 키에 연결할 수 있다.

표준 키 매핑 기준:

- 3개 이상 사업에서 반복될 가능성이 높다.
- 제출 의미가 사업마다 동일하다.
- 사용자가 다른 사업에서도 같은 문서로 이해할 수 있다.

표준 키 예시:

- `tax_invoice`: 세금계산서
- `credit_card_receipt`: 신용카드 영수증
- `contract`: 계약서
- `quote`: 견적서
- `comparative_quote`: 비교견적서
- `vendor_business_registration`: 거래처 사업자등록증
- `vendor_bankbook_copy`: 거래처 통장사본
- `inspection_report`: 검수조서 또는 검수확인서
- `result_report`: 결과보고서

사업별 전용 키 생성 기준:

- 특정 사업, 기관, 양식, 제출 맥락에 종속된다.
- 명칭이 같아도 사업별 작성 양식이나 의미가 다를 수 있다.
- 표준 키에 억지로 맞추면 추적성이 떨어진다.

사업별 전용 키 예시:

- `expense_payment_request`: 사업비 지급요청서
- `material_usage_plan`: 재료활용계획서
- `equipment_usage_plan`: 기계장치활용계획서
- 사업 전용 확인서류 또는 승인 공문

관리자 확인 필요 기준:

- 이름은 비슷하지만 제출 의미가 애매하다.
- `필요 시`, `해당 시`, `증빙사진 포함`처럼 조건 또는 첨부 관계가 문맥에 따라 달라진다.
- 자동 추출 결과가 문서 표 구조와 일치하는지 확신하기 어렵다.

## 8. 원문 보존

자동 추출과 관리자 편집 이후에도 원문 근거를 보존한다.

정책 항목은 최소한 아래 정보를 가진다.

```ts
type PolicySourceReference = {
  document_id: string;
  original_file_name: string;
  page_number: number | null;
  table_index: number | null;
  row_index: number | null;
  raw_category_name: string | null;
  raw_subcategory_name: string | null;
  raw_evidence_text: string;
};
```

화면에는 정제된 이름을 표시하더라도, 추출 원문과 PDF 위치는 검수 및 추적을 위해 유지한다.

## 9. 버전 규칙

- `confirmed` 정책은 불변이다.
- 수정이 필요하면 새 초안 버전을 만든다.
- 새 버전도 검수와 확정 절차를 거쳐야 한다.
- 프로젝트는 하나의 `confirmed` 정책 버전을 참조한다.
- 지출은 생성 또는 비목 변경 시점의 정책 버전과 증빙 요구사항 스냅샷을 참조한다.
- 이전 정책은 `archived` 상태로 바꿀 수 있지만 기존 지출 기록에서는 계속 조회 가능해야 한다.
- 정책 PDF는 사업 등록 시 첨부할 수 있고, 사업 운영 중 사업 관리 화면에서도 새 버전으로 추가 업로드할 수 있다.
- 새 정책 PDF 업로드는 기존 `confirmed` 정책을 직접 바꾸지 않고 새 초안 버전을 만든다.
- 새 정책 버전이 확정되어도 기존 지출의 정책 스냅샷은 자동 변경하지 않는다.
- 기존 지출을 새 정책 버전으로 갱신하는 기능은 사용자가 명시적으로 선택하는 후속 작업으로 다룬다.

## 10. Fallback 동작

정책 파일을 업로드하지 않은 사업은 현재 제품 동작을 유지한다.

- 기존 비목 선택 UI를 유지한다.
- 기존 지출 카드 레이아웃을 유지한다.
- 기존 증빙 업로드 정책을 유지한다.
- 사업별 정책 안내나 하위 유형 선택은 표시하지 않는다.

이때 사용하는 기존 비목 목록은 `legacy fallback category set`으로만 취급한다.

- legacy fallback category set은 정책 파일이 없거나, 정책이 미확정이거나, 추출 실패 상태인 사업에서만 사용한다.
- 확정 정책이 있는 사업에는 legacy fallback category set을 섞지 않는다.
- legacy fallback category set은 새 사업별 정책 초안의 기본값, 병합 대상, 누락 항목 보강 기준으로 사용하지 않는다.
- 정책 기반 운영으로 모든 사업이 전환된 뒤에는 별도 마이그레이션으로 제거할 수 있다.

정책 파일이 업로드되었지만 아직 확정되지 않은 경우도 fallback UI를 유지한다.
다만 사업 관리 화면에서는 정책 초안 상태와 검수 필요 안내를 표시한다.
지출 등록 화면과 지출 상세 화면에는 낮은 강도의 안내 배너만 표시한다.

정책 미확정 안내 규칙:

- 지출 등록/상세 화면에는 경고나 오류 톤을 사용하지 않는다.
- 안내 문구는 `정책 검수 전이라 기존 비목 기준으로 운영 중입니다.` 수준으로 표시한다.
- 안내 배너에는 `정책 검수로 이동` CTA만 제공한다.
- 정책 편집, key 수정, 확정 작업은 지출 화면 안에서 열지 않고 사업 관리 화면에서 수행한다.
- 정책이 확정되면 안내 배너는 사라지고 정책 기반 비목/증빙 목록이 적용된다.

PDF 자동 추출이 실패하거나 추출 품질이 낮아 정책 초안을 만들기 어려운 경우도 fallback UI를 유지한다.
1차에서는 추출 실패 상태에서 관리자가 빈 정책을 처음부터 수동 생성하는 흐름을 제공하지 않는다.
추출 실패한 primary PDF는 근거 문서로 보존하고, 사업 관리 화면에서 추출 실패 상태와 재업로드 안내를 표시한다.
해당 사업은 새 정책이 확정될 때까지 기존 비목 선택 방식, 기존 지출 카드 레이아웃, 기존 증빙 업로드 정책으로 운영한다.
추출 실패 상태의 지출 등록/상세 화면에는 작은 안내 배너에 `재업로드 필요` 문구를 함께 표시하고, 재업로드는 사업 관리 화면에서만 진행한다.

## 10-1. 정책 업로드와 교체 위치

정책 PDF는 사업 등록 흐름과 사업 관리 화면에서 모두 업로드할 수 있다.

사업 등록 시:

- 정책 PDF 첨부는 선택이다.
- 정책 PDF를 첨부하지 않아도 사업 등록은 완료할 수 있다.
- 첨부된 정책 PDF는 정책 초안 생성 대상으로 등록한다.
- 정책이 확정되기 전까지는 기존 비목/지출 카드 레이아웃을 유지한다.

사업 관리 화면:

- 기존 사업에 정책 PDF를 나중에 추가할 수 있다.
- 확정 정책이 이미 있는 사업에도 새 정책 PDF를 업로드해 새 버전 초안을 만들 수 있다.
- 새 버전 초안은 검수/확정 전까지 기존 확정 정책을 대체하지 않는다.
- 새 버전이 확정되면 이후 새 지출 또는 비목 변경에 사용할 수 있다.
- 기존 지출에 새 정책을 적용하는 동작은 자동 수행하지 않는다.

## 10-2. 정책 버전별 문서 구성

1차에서는 하나의 정책 버전마다 자동 추출 대상 문서를 1개로 제한한다.

```ts
type ProgramPolicyDocumentRole = "primary" | "reference";
```

- `primary`: 해당 정책 버전의 자동 추출 기준 문서다.
- `reference`: 관리자 검수 시 참고할 수 있는 보조 근거 문서다.

1차 규칙:

- 하나의 정책 버전에는 `primary` 문서가 정확히 1개 있어야 한다.
- 자동 추출은 `primary` 문서에만 수행한다.
- `reference` 문서는 여러 개 첨부할 수 있다.
- `reference` 문서는 자동 병합하거나 자동 정책 항목으로 만들지 않는다.
- 관리자는 `reference` 문서를 보며 초안 항목을 수동으로 보정할 수 있다.
- `reference` 문서에서 수동 보정한 항목도 원문 파일명, 페이지, 원문 문구를 보존한다.

여러 PDF를 자동 병합해 하나의 정책 초안을 만드는 기능은 1차 범위에서 제외한다.

## 11. MVP 제외 범위

- PDF 내용만으로 정책을 자동 확정하지 않는다.
- OCR 기반 이미지 PDF 인식은 필수 범위가 아니다.
- 여러 PDF를 자동 병합해 정책 초안을 생성하지 않는다.
- PDF 추출 실패 상태에서 빈 정책을 처음부터 수동 생성하지 않는다.
- 증빙 파일의 진위 여부를 자동 검증하지 않는다.
- 증빙 누락 여부로 지출 저장이나 단계 이동을 차단하지 않는다.
- 외부 기관 시스템과 연동하지 않는다.
- 여러 정책 버전을 동시에 한 프로젝트에 적용하지 않는다.

## 12. Acceptance Criteria

- 정책 파일이 없는 사업은 기존 비목과 지출 카드 레이아웃을 유지한다.
- 정책 파일이 있는 사업은 PDF 추출을 시도하고 정책 초안을 만든다.
- 확정 전 정책은 지출 카드에 자동 적용되지 않는다.
- 관리자는 비목, 하위 유형, 증빙 요구사항, 표준 키 매핑을 검수하고 수정할 수 있다.
- 확정된 정책만 신규 지출의 사업별 비목 선택과 증빙 목록에 적용된다.
- 하위 유형이 있는 비목은 상위 비목과 하위 유형을 함께 선택할 수 있다.
- 지출 증빙 목록은 상위 비목 공통 증빙과 하위 유형 증빙을 병합해 표시한다.
- 확정 정책 수정은 새 버전을 통해서만 가능하다.
- 지출은 적용 당시 정책 버전과 증빙 요구사항 스냅샷을 추적할 수 있다.
- 원문 PDF 파일명, 페이지, 표/행 위치, 원문 문구가 보존된다.

## 13. 1차 커스텀 대시보드 범위

이 기능은 사업별 커스텀 운영 대시보드의 첫 기반이다.
다만 1차 범위는 아래로 제한한다.

포함:

- 사업별 정책 PDF 기반 비목 생성
- 상위 비목과 선택적 하위 유형 생성
- 상위 비목 공통 증빙과 하위 유형 증빙 생성
- 증빙 문서의 표준 키 또는 사업별 전용 키 매핑
- 확정된 사업별 비목 구조를 대시보드 카테고리 그룹에 반영
- 확정된 사업별 비목 구조를 지출 필터에 반영

제외:

- 사업별 KPI 공식 커스텀
- 사업별 칸반 단계명 또는 단계 수 커스텀
- 사업별 지출 단계 이동 규칙 커스텀
- 사업별 자동 적합성 점수 또는 risk score 계산
- 기관 시스템 연동

1차에서는 단계명, KPI, 칸반 구조는 현재 제품의 공통 구조를 유지한다.
사업별 차이는 비목, 하위 유형, 증빙 요구사항, 그리고 그에 따른 그룹/필터 표현에만 적용한다.

1차 대시보드 표현:

- 대시보드 카테고리 그룹은 상위 비목 기준으로 유지한다.
- 하위 유형은 독립 그룹으로 승격하지 않는다.
- 하위 유형은 지출 카드 또는 지출 행의 보조 라벨로 표시한다.
- 하위 유형은 필터 옵션으로 사용할 수 있다.
- CSV 내보내기 반영은 1차 범위에서 제외하고 후순위 백로그로 둔다.
- 하위 유형이 없는 지출은 하위 유형 컬럼을 비워 둔다.

## 14. PDF 형식 차이 리스크

기관별 PDF는 표 구조와 표현 방식이 다를 수 있다.
샘플 확인 기준으로 다음 차이를 허용해야 한다.

- 같은 비목/증빙 정보가 여러 페이지에 나뉘어 있을 수 있다.
- 상위 비목과 하위 유형이 별도 표로 분리될 수 있다.
- 상위 비목과 하위 유형이 한 셀 안의 불릿으로 섞여 있을 수 있다.
- 표 헤더가 `비목 / 집행 증빙서류`가 아닐 수 있다.
- `비목 정의`처럼 보이는 헤더 아래에 실제 증빙 목록이 들어갈 수 있다.
- `공통제출서류`, `필수`, `필요 시`, `해당 시` 같은 문구가 비목명, 조건, 안내문 사이에서 섞일 수 있다.
- `증빙사진 포함`처럼 독립 증빙이 아니라 다른 증빙에 붙는 첨부 조건이 있을 수 있다.

따라서 PDF 파서는 추출 결과를 곧바로 확정 정책으로 보지 않는다.
추출 결과는 관리자 검수를 위한 초안이며, 초안 항목마다 검수 상태를 가져야 한다.

```ts
type ExtractedPolicyItemReviewStatus =
  | "auto_confident"
  | "needs_admin_review"
  | "manual_required";
```

- `auto_confident`: 표 구조와 항목 구분이 비교적 명확한 자동 추출 항목이다.
- `needs_admin_review`: 추출은 되었지만 조건, 하위 유형, 병합 여부 확인이 필요하다.
- `manual_required`: 자동 추출만으로 정책 항목을 만들기 어렵다.

사용 가능한 초안의 최소 기준은 아래와 같다.

- 상위 비목이 1개 이상 추출되어야 한다.
- 추출된 각 상위 비목에는 화면 표시명과 stable key를 만들 수 있어야 한다.
- 최소 1개 이상의 증빙 요구사항이 특정 상위 비목 또는 공통 증빙으로 연결되어야 한다.

위 기준을 만족하지 못하면 정책 초안을 억지로 만들지 않는다.
primary PDF와 추출 실패 정보를 보존하고, 해당 사업은 정책이 확정될 때까지 기존 비목 선택과 지출 카드 레이아웃을 유지한다.

## 15. 정책 확정 조건

정책은 아래 조건을 만족해야 `confirmed`로 전환할 수 있다.

- 모든 상위 비목에 stable key와 화면 표시명이 있다.
- 하위 유형이 있는 비목은 공통 증빙과 하위 유형 증빙이 분리되어 있다.
- 모든 증빙 요구사항에 stable key, 표시명, 요구 유형, 충족 방식이 있다.
- 조건부 증빙은 조건 문구가 보존되어 있다.
- 원문 PDF 파일명, 페이지, 표/행 위치, 원문 문구가 보존되어 있다.
- `needs_admin_review` 또는 `manual_required` 항목이 남아 있지 않다.
- 정책 적용 미리보기에서 지출 등록 비목 선택, 대시보드 그룹, 필터 기준을 확인할 수 있다.

## 15-1. 정책 확정 권한과 기록

1차에서는 별도 역할/승인 워크플로우를 만들지 않는다.
현재 로그인한 내부 사용자가 정책을 확정할 수 있다.

정책 확정은 이후 지출 등록, 지출 상세, 대시보드 그룹, 필터 기준에 영향을 주는 무거운 액션이다.
따라서 확정 전에는 체크리스트와 확인 모달을 제공한다.

확정 전 체크리스트:

- 검수 필요 항목이 0개다.
- 상위 비목 수를 확인했다.
- 하위 유형 수를 확인했다.
- 증빙 요구사항 수를 확인했다.
- 추출 실패 또는 수동 필요 항목이 없다.
- 확정 후 직접 수정할 수 없고 새 버전으로만 변경할 수 있음을 확인했다.

확정 시 기록할 값:

```ts
type PolicyConfirmationMetadata = {
  confirmed_by: string;
  confirmed_at: string;
  confirmed_summary: {
    category_count: number;
    subcategory_count: number;
    evidence_requirement_count: number;
  };
};
```

`confirmed_by`는 현재 로그인 사용자 ID를 기록한다.
향후 다중 사용자/역할 기능이 도입되면 별도 승인 권한 정책을 추가한다.

## 16. 1차 관리자 검수 범위

1차 구현의 핵심은 PDF 자동화율이 아니라, 자동 추출 초안을 관리자가 신뢰 가능한 정책 데이터로 확정할 수 있는 틀을 만드는 것이다.

관리자는 1차에서 최소한 아래 항목을 수정할 수 있어야 한다.

- 상위 비목 표시명과 stable key
- 하위 유형 표시명과 stable key
- 증빙 문서 표시명과 stable key
- 증빙 요구 유형: `required`, `conditional`, `optional`
- 충족 방식: `any_of`, `all_of`
- 조건 문구
- 원문 PDF 매핑: 파일명, 페이지, 표/행 위치, 원문 문구
- 항목 검수 상태

1차에서는 복잡한 조건식을 만들지 않는다.
예를 들어 `2천만원 초과 시`, `카드결제 시 생략가능`, `필요 시` 같은 조건은 자동 계산식이 아니라 조건 문구로 보존하고 사용자에게 안내한다.

1차 정책 검수 화면은 행 단위 표 편집으로 제공한다.

- 상위 비목, 하위 유형, 증빙 요구사항을 각각 행 단위로 수정, 추가, 삭제할 수 있다.
- 복잡한 트리 드래그 편집이나 시각적 구조 편집은 1차에서 제외한다.
- 하위 유형은 상위 비목 행 또는 별도 하위 유형 표에서 상위 비목 key로 연결한다.
- 증빙 요구사항은 상위 비목 key와 선택적 하위 유형 key로 연결한다.
- 검수 화면은 정책 편집의 중심 화면이며, 지출 등록/상세 화면 안에서는 정책 편집 UI를 열지 않는다.

PDF 원문 근거는 각 추출 항목 옆에서 확인할 수 있어야 한다.

- 각 항목에는 원문 파일명, 페이지, 표/행 위치, 원문 문구를 표시한다.
- 1차에서는 화면 안 PDF 뷰어를 내장하지 않는다.
- PDF 원문은 다운로드 또는 새 화면/새 탭 열기 수준으로 제공한다.
- 원문 근거가 없는 자동 추출 항목은 확정할 수 없다.

정책 확정 차단 오류는 최소 오류 세트로 제한한다.
아래 오류가 남아 있으면 `confirmed`로 전환할 수 없다.

- 상위 비목, 하위 유형, 증빙 요구사항의 stable key가 누락되었거나 중복되었다.
- 화면 표시명이 누락되었다.
- 증빙 요구사항의 요구 유형 또는 충족 방식이 누락되었다.
- 원문 근거 파일명, 페이지 또는 원문 문구가 누락되었다.
- `needs_admin_review` 또는 `manual_required` 상태 항목이 남아 있다.

조건 문구의 의미 해석, 금액 기준 판단, 결제수단별 예외 판단은 1차 확정 차단 기준에 포함하지 않는다.

## 17. 단계적 고도화 원칙

1차는 전체 정책 데이터 모델과 검수/확정 흐름을 먼저 안정화한다.
자동 추출 정확도와 자동 조건 판단은 이후 단계에서 고도화한다.

고도화 순서:

```text
1차: 정책 초안 생성, 관리자 검수/확정, 확정 정책의 지출/대시보드/필터 반영
2차: 기관별 PDF 형식별 추출 규칙 개선
3차: 반복 문서의 표준 키 자동 매핑 정확도 개선
4차: 조건부 증빙 자동 활성화
5차: 사업별 KPI, 단계, 검증 로직 커스텀
```

1차에서 하지 않는 것:

- PDF 자동 추출 결과를 검수 없이 확정하지 않는다.
- PDF 추출 실패 상태에서 빈 정책을 처음부터 수동 생성하지 않는다.
- 조건부 증빙을 금액, 결제수단, 거래처 정보로 자동 활성화하지 않는다.
- 사업별 KPI 공식을 변경하지 않는다.
- 사업별 칸반 단계명이나 단계 수를 변경하지 않는다.
- 사업별 단계 이동 규칙을 변경하지 않는다.
- 증빙 누락으로 지출 저장이나 단계 이동을 차단하지 않는다.

## 18. 정책 확정 전 미리보기

1차 정책 확정 전 미리보기는 실제 화면 전체 재현이 아니라 정책 적용 결과 요약으로 제공한다.

미리보기에서 보여줄 것:

- 지출 등록 시 표시될 상위 비목 목록
- 하위 유형이 있는 비목 목록
- 비목/하위 유형별 필수, 조건부, 선택 증빙 개수
- 대시보드 상위 비목 그룹 순서
- 필터에 노출될 상위 비목과 하위 유형 목록

미리보기에서 제외할 것:

- 실제 대시보드 전체 렌더링
- 실제 지출 카드 전체 UI 렌더링
- 실제 CSV 파일 생성 또는 CSV 컬럼 미리보기

CSV 내보내기 반영은 1차 정책 기능에서 고려하지 않는다.
CSV 정책 반영은 별도 백로그 기능으로 다룬다.

## 19. 정책 운영 상태와 1차 완료 기준

사업 관리 화면에는 현재 사업의 정책 운영 상태를 명확히 표시한다.

```ts
type ProjectPolicyOperationStatus =
  | "legacy_fallback"
  | "draft_needs_review"
  | "confirmed_policy"
  | "extraction_failed";
```

- `legacy_fallback`: 정책 PDF가 없거나 확정 정책이 없어 기존 비목/지출 카드 흐름으로 운영 중이다.
- `draft_needs_review`: 정책 초안이 생성되었고 관리자 검수 또는 확정이 필요하다.
- `confirmed_policy`: 확정 정책이 있으며 새 지출과 비목 변경에 정책 기준을 적용한다.
- `extraction_failed`: primary PDF에서 사용 가능한 초안을 만들지 못해 재업로드 또는 기존 흐름 유지가 필요하다.

정책 확정 후 기존 지출은 자동 갱신하지 않는다.

- 기존 지출의 정책 스냅샷은 생성 또는 비목 변경 시점 기준으로 유지한다.
- 정책 확정 전 legacy 기준으로 생성된 지출은 상세 화면에서 이전 기준으로 생성된 지출임을 낮은 강도로 안내한다.
- 기존 지출을 새 정책 버전으로 일괄 재적용하는 기능은 1차 범위에서 제외하고 후순위로 둔다.

정책 버전 화면은 1차에서 목록과 요약만 제공한다.

- 버전 번호 또는 생성 순서
- 상태
- primary PDF 파일명
- 확정일
- 확정자
- 상위 비목, 하위 유형, 증빙 요구사항 개수

버전 간 diff 비교 UI는 1차 범위에서 제외한다.

추출 실패 상태에서 제공하는 관리자 액션은 아래로 제한한다.

- primary PDF 재업로드
- 원문 PDF 확인 또는 다운로드
- 기존 비목/지출 카드 흐름 계속 사용

추출 실패 상태에서 빈 정책을 처음부터 수동 생성하는 기능은 제공하지 않는다.

1차 완료 기준은 아래 end-to-end 흐름이 동작하는 것이다.

```text
정책 PDF 업로드
-> 정책 초안 생성 또는 추출 실패 fallback
-> 행 단위 검수/편집
-> 정책 확정
-> 새 지출 등록 시 정책 비목/하위 유형/증빙 목록 표시
-> 대시보드 상위 비목 그룹과 필터에 정책 구조 반영
```

이 흐름 밖의 자동 검증, 고급 PDF 뷰어, 버전 diff, 기존 지출 일괄 재적용은 후순위로 둔다.

## 20. 초기 도입 데이터 초기화 기준

현재 제품은 pre-MVP/개발 초기 단계이므로, 1차 정책 기능 도입 과정에서 기존 지출 데이터를 보존해야 하는 제약은 없다.
이 기준은 정식 운영 데이터가 생기기 전까지만 유효하다.

1차 구현 마이그레이션에서 초기화할 수 있는 데이터:

- 기존 지출
- 기존 프로젝트 비목
- 기존 지출 증빙 연결
- 기존 지출 증빙 파일 metadata
- 기존 지출 기준 대시보드 집계 데이터

1차 구현 마이그레이션에서 보존해야 하는 데이터:

- 회사 기본 정보
- 사업 기본 정보
- 사업 등록 시 첨부한 정책 PDF
- 정책 버전 데이터
- 확정된 정책과 정책 원문 근거

운영 전환 이후에는 이 destructive reset 기준을 폐기한다.
정식 운영 데이터가 생긴 뒤에는 기존 지출과 증빙을 보존하는 비파괴 마이그레이션 정책을 별도로 수립해야 한다.
